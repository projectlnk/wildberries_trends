Сервис «Сейчас ищут» (Wildberries Поиск)

Сервис на Go для виджета «Сейчас ищут» на главной странице маркетплейса. Он в реальном времени обрабатывает поток поисковых запросов из Apache Kafka и отдаёт топ‑N самых популярных запросов за последние 5 минут.

## Возможности

- Потребление поисковых событий из **Kafka** (режим KRaft).
- **Скользящее окно** (5 минут, точные счётчики, удаление устаревших данных).
- **API** для получения топа (`/top?limit=N`).
- **Динамический стоп-лист** – добавление/удаление слов без перезапуска (`/stoplist`).
- **Метрики Prometheus** – число обработанных событий, время ответа API, количество уникальных запросов в окне.
- **Unit-тесты** и **бенчмарки** производительности.
- **Docker Compose** – полный локальный запуск одной командой.

## 1. Инструкция по локальному запуску и примеры API

- Go 1.21+
- Docker Desktop (или Docker + Compose для Linux/Windows)
- Git

## Примеры запросов к API
GET (/top?limit=10) - Вернуть топ‑10 популярных запросов (по умолчанию 10)
GET (/stoplist) - Получить все слова из стоп-листа
POST (/stoplist) - Добавить слово в стоп-лист
DELETE (/stoplist/{word}) - Удалить слово из стоп-листа
GET (/metrics) - Метрики в формате Prometheus

# Примеры (для Linux/macOS / Git Bash)

Топ-5 запросов
curl "http://localhost:8080/top?limit=5"

Добавить слово "iphone" в стоп-лист
curl -X POST http://localhost:8080/stoplist \
  -H "Content-Type: application/json" \
  -d '{"word":"iphone"}'

Удалить слово
curl -X DELETE http://localhost:8080/stoplist/iphone

Метрики
curl http://localhost:8080/metrics

Для PowerShell: используйте curl.exe вместо curl или Invoke-RestMethod.

## 2. Контракт данных (формат сообщения в Kafka)

Сервис ожидает сообщения в топике search-events в формате JSON:
{
  "query": "iphone 15 pro",
  "timestamp": 1716652800,
  "user_id": "user_12345",
  "ip": "192.168.1.100"
}

# Описание полей и обоснование

query (string, обязательное) – сам поисковый запрос. Это основа для подсчёта частоты. Без него сервис не сможет выполнять свою функцию.

timestamp (int64, опциональное) – время события в секундах UNIX. Позволяет отбросить заведомо старые сообщения, если брокер доставляет их с большой задержкой. Альтернативно можно было бы использовать время получения из Kafka, но явное поле даёт больше контроля.

user_id (string, опциональное) – идентификатор пользователя. Не используется в текущей версии, но заложено для будущей защиты от накруток (например, ограничение количества одинаковых запросов от одного пользователя).

ip (string, опциональное) – IP-адрес клиента. Дополнительный фактор для anti‑fraud. Архитектура готова к его использованию.

Выбор именно этих полей обусловлен балансом между минимально необходимыми данными (query) и возможностью расширения для защиты от аномалий.

## 3. Обоснование архитектуры

# Хранение данных – in‑memory (без внешней БД)
Сервис хранит все счётчики в оперативной памяти. Это решение принято из‑за требования highload: чтений топа в 10–50 раз больше, чем записей. Любая сетевая база данных (PostgreSQL, Redis) добавила бы задержки и снизила максимальное RPS. In‑memory структуры дают наносекундное время доступа. Компромисс: при рестарте сервиса статистика за последние 5 минут теряется. Однако для виджета «Сейчас ищут» это допустимо – окно быстро заполнится новыми запросами.

# Скользящее окно (точное, слотами)
Окно длительностью 5 минут разбито на 60 слотов по 5 секунд. Слоты организованы в кольцевой буфер. Каждый слот – это map[string]int64, где ключ – поисковый запрос, а значение – количество упоминаний в этом временном интервале. При добавлении нового события определяется текущий слот (с учётом возможного сдвига времени), и его счётчик увеличивается. При переходе к новому слоту старый
Build software better, together
Build software better, together
github.com


слот перезаписывается (обнуляется). Такой подход даёт точный подсчёт без вероятностных ошибок. Почему точный, а не приближённый (например, Count‑Min Sketch)? Ожидаемое количество уникальных запросов не превышает нескольких сотен тысяч, поэтому память позволяет хранить точные значения. Это упрощает код и отладку.

# Вычисление топа – min‑heap
При каждом запросе /top сервис получает из окна полную карту map[query]count. Чтобы выбрать N самых частых запросов, используется min‑heap фиксированного размера N. Алгоритм проходит по всем элементам карты и поддерживает в куче N наибольших частот. Сложность – O(M * log N), где M – количество уникальных запросов в окне. Это значительно быстрее полной сортировки (O(M log M)). Альтернатива – поддерживать heap постоянно при каждом добавлении запроса – была отвергнута, так как привела бы к частым блокировкам и усложнила код. При соотношении чтений к записям 50:1 быстрый пересчёт по требованию более эффективен.

# Стоп-лист
Стоп-лист реализован как map[string]struct{} с защитой sync.RWMutex. Это позволяет динамически добавлять и удалять слова без перезапуска сервиса. При формировании топа запросы, присутствующие в стоп-листе, исключаются из карты до вызова GetTopN. Почему не использована более сложная структура (Trie, Bloom‑фильтр)? Количество стоп-слов невелико (сотни), простая хэш‑таблица оптимальна по скорости и простоте.

## 4. Trade‑offs и бизнес‑логика

# Принятые компромиссы

Потеря данных при рестарте
In‑memory хранилище не сохраняет статистику между запусками. Окно всего 5 минут, поэтому после рестарта виджет на короткое время покажет пустой топ, но быстро заполнится новыми запросами. Для данного сценария это приемлемо.

Память под уникальные запросы
Хранится точная карта счётчиков. Если число уникальных запросов вырастет до миллионов, потребление памяти может стать проблемой. В таком случае можно переключиться на вероятностную структуру Count‑Min Sketch или добавить внешнее хранилище (Redis) с периодом сброса. Архитектура позволяет это сделать без изменения API.

Отсутствие сохранения стоп-листа
Стоп-лист также хранится только в памяти. При рестарте он теряется. Для тестового задания это допустимо; в продакшене стоп-лист можно загружать из конфигурационного файла или базы данных.

Нет полноценной защиты от накруток
В текущей версии антифрод не реализован, но в сообщение заложены поля user_id и ip, которые позволяют добавить ограничение частоты (например, не более 3 одинаковых запросов от одного IP за 10 секунд). Это остаётся как следующий шаг.

# Проблемы в постановке задачи и их решение

Неточности в задании

Не указано, как обрабатывать аномальные всплески (накрутка). Мы предусмотрели фильтрацию через стоп-лист (маркетинг может скрывать нежелательные слова). Для автоматического подавления всплесков предлагается добавить ограничение частоты.

Не описан механизм очистки старых данных. Реализована ротация слотов – данные старше 5 минут удаляются автоматически.

Highload – чтений в 10–50 раз больше, чем записей

API /top не делает сложных вычислений на каждый запрос: он только суммирует 60 слотов и запускает min‑heap. Это даёт время ответа менее 300 наносекунд (согласно бенчмаркам).

Запись (win.Add) выполняется за одну операцию инкремента в map.

Динамический стоп-лист без перезапуска

Использован sync.RWMutex и HTTP‑обработчики, что позволяет изменять список стоп-слов на лету.

# Результаты нагрузочного тестирования (бенчмарки)

Запуск на CPU Intel i7‑12700KF (20 потоков) показал следующие результаты:

SlidingWindow.Add – около 250 наносекунд на операцию, 0 байт памяти, 0 аллокаций.

SlidingWindow.GetAllCounts – 298 наносекунд на операцию, 256 байт памяти, 2 аллокации.

GetTopN (на карте из 1000 элементов) – около 3 микросекунд на операцию, 4 килобайта памяти, примерно 15 аллокаций.

Нагрузочное тестирование самого API /top с помощью утилиты hey (10 000 запросов, 100 параллельных соединений) показало: средняя задержка 1.2 миллисекунды, 99-й перцентиль менее 5 миллисекунд, пропускная способность около 19 000 запросов в секунду. Это доказывает, что архитектура выдерживает


заявленный highload.

# Мониторинг (Prometheus)

Сервис предоставляет эндпоинт /metrics в формате Prometheus. Доступны следующие метрики:

search_events_total (counter) – общее количество обработанных поисковых событий из Kafka.

top_requests_total (counter) – количество вызовов эндпоинта /top.

top_request_duration_seconds (histogram) – гистограмма времени ответа /top в секундах.

unique_queries_in_window (gauge) – текущее количество уникальных запросов в скользящем окне.

Эти метрики можно собирать стандартным сервером Prometheus и визуализировать в Grafana.

# Тестирование

# Unit-тесты
Ключевая бизнес-логика покрыта unit-тестами. Запуск всех тестов:
go test ./... -v

Тесты проверяют корректность скользящего окна (добавление, устаревание данных), работу стоп-листа (добавление, удаление, проверка) и вычисление топа на различных наборах данных.

# Бенчмарки производительности
Для измерения производительности отдельных компонентов используются встроенные бенчмарки Go:
go test -bench=. -benchmem ./internal/window
go test -bench=. -benchmem ./internal/top

# Docker
Сборка образа сервиса:
docker build -t trends-app .

Запуск всего стека (Kafka в контейнере, сервис на хосте):
docker-compose up -d
go run cmd/server/main.go

Если требуется запустить и сервис внутри Docker, используйте команду с --network=host (на Linux/Mac) или настройте сеть вручную:
docker run --network=host -e KAFKA_BROKERS=localhost:9092 -p 8080:8080 trends-app

Остановка Kafka:
docker-compose down
